Instruksi ini diterbitkan untuk merombak total aspek kenyamanan antarmuka (UI) dan pengalaman pengguna (UX) pada aplikasi SISDMK. Seluruh pengembang wajib mengikuti standarisasi tata letak, komponen interaktif, serta optimasi mobile yang tercantum di bawah ini guna memastikan sistem nyaman digenggam, inklusif, dan meminimalisir kesalahan operasional birokrasi.

1. Standarisasi Token Desain Visual Global

Seluruh pengembang wajib menyamakan variabel warna dan tipografi pada berkas konfigurasi Tailwind CSS untuk menjaga konsistensi identitas visual Dinas Kesehatan Provinsi DKI Jakarta.

Konfigurasi Warna dan Tipografi (tailwind.config.ts):

const config = {
  theme: {
    extend: {

  }
};
export default config;

2. Rekayasa Ulang Komponen dan Refaktorisasi Kode Modul Utama

A. Refaktorisasi Tabel Pegawai Menjadi Responsive Card Toggle (Modul F-04 & NF-02)

Dilarang keras membiarkan tabel data pegawai mengalami overflow horizontal pada perangkat mobile. Terapkan sistem penyembunyian responsif menggunakan breakpoint Tailwind CSS.

Resolusi Layar Komponen UI yang Wajib Aktif Spesifikasi UX
Layar Kecil (< 768px) / Smartphone Card Layout List Format satu kolom vertikal, sudut rounded-xl, efek active:scale-[0.99] saat ditekan oleh jempol pengguna.
Layar Besar (≥ 768px) / Desktop Standard Data Table Server-side pagination, sorting indikator transparan, limit baris per halaman. 
// Implementasi pada src/components/pegawai/EmployeeList.tsx

B. Penguatan Kontrol Mekanisme Form Bertahap & Auto-Save Draft (Modul F-06 & NF-03)

Untuk mengatasi kelelahan pengisian formulir data pribadi, alamat, keluarga, dan riwayat, form wajib dibagi menggunakan sistem visual stepper didukung penyimpanan lokal (localStorage).

  1. Setiap input wajib memiliki Instant Inline Validation (contoh: NIK langsung memunculkan peringatan kuning jika jumlah digit tidak tepat sama dengan 16 angka).
  2. Gunakan Sticky Footer Action Bar pada resolusi mobile agar tombol "Selanjutnya" dan "Simpan Draft" selalu menetap di area jangkauan jempol.
  3. Terapkan dropdown pencarian teks terintegrasi (*Searchable Dropdown*) untuk memetakan referensi wilayah kelurahan, kecamatan, dan kota guna memangkas durasi gulir layar.

C. Pembuatan Indikator Responsif & Skeleton Screen pada Modul Import DRH (Modul F-08 & NF-03)

Proses parsing dokumen PDF DRH memakan waktu pemrosesan background query yang intensif. Pengembang dilarang menggunakan animasi loading berputar statis (spinner) tanpa kejelasan informasi waktu.

  • Wajib Menggunakan Progress Bar Persentase: Tampilkan pergerakan angka persentase kelancaran proses parsing file.
  • Wajib Menggunakan Skeleton Screen Loader: Tampilkan blok layout bayangan yang berkedip (*animate-pulse*) pada area preview transaksi data untuk meredam kecemasan tunggu pengguna.

D. Layout Split-Screen untuk Validasi Berkas Usulan (Modul F-10, F-11 & F-12)

3. Protokol Validasi Kenyamanan pada Tahap UAT

Sebelum aplikasi dinyatakan siap digunakan secara operasional (Go-Live), tim pengembang bersama tim penjamin mutu wajib menguji aspek kebergunaan antarmuka menggunakan kuesioner **System Usability Scale (SUS)** kepada pengguna perwakilan dari 3 level peran:

  1. Admin UKPD: Diuji ketepatan pengisian formulir bertahap menggunakan gawai smartphone.
  2. Admin Wilayah: Diuji kecepatan waktu eksekusi verifikasi berkas usulan menggunakan skema layar terbagi.
  3. Super Admin / Dinas Pusat: Diuji fleksibilitas penarikan data drilldown pada sebaran dashboard analitik SDM.

Ambang batas kelulusan nilai rata-rata pengujian SUS untuk UI/UX baru ini ditetapkan minimal berada pada skor 75 (Kategori Good / Acceptable) sebelum diizinkan migrasi ke server produksi utama.
